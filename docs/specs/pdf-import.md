# PDF取込仕様

## 方針

初期対応は楽天カード明細PDFを対象とする。

今後、カード会社や明細フォーマットを少しずつ追加できるように、PDF解析はフォーマットごとの抽出ルールとして拡張できる設計にする。

## 初期対象

- 楽天カード明細PDF

## 抽出方式

アップロードされた明細PDFは、テーブルの列が固定されている前提でルールベース抽出する。

PDF抽出ライブラリは PyMuPDF を使う。

想定列:

- 利用日
- 利用店名
- 利用者
- 支払方法
- 利用金額

## 取込データへのマッピング案

| PDF列 | 明細項目 |
| --- | --- |
| 利用日 | 日付 |
| 利用店名 | 店名 |
| 支払方法 | 支払い方法 |
| 利用金額 | 金額 |
| 利用者 | 利用者 |

初回取込時のカテゴリは未分類とし、ユーザーが分類する。

次回以降は同一項目の過去分類からルールベースで自動分類する。過去分類が見つからない場合は未分類にする。

支払い方法は楽天PDFの値をそのまま保存する。

利用金額がマイナスの場合は取消明細として取り込む。

0円明細を許可する。

アップロード済みPDFは原本として保存する。

初期のPDF原本保存先はローカルの `storage/uploads/` とする。保存パスは `storage/uploads/{user_id}/{upload_id}/original.pdf` の相対パスとする。将来はS3互換ストレージへの移行を想定する。

PDF原本はユーザーが削除するまで保存する。

アップロード履歴を論理削除したとき、保存済みPDF原本も即削除する。

最大アップロードサイズは10MBとする。

重複判定キーは以下を基本とする。

- ユーザーID
- 利用日
- 利用店名
- 利用者
- 支払方法
- 利用金額
- `source_file_name`
- `source_row_number`
- `source_page_number`
- `source_format`
- `source_hash`

楽天カード明細PDFのテストデータは、実PDFではなく抽出後テキストfixtureと期待値JSONで管理する。

実装では `POST /api/uploads` でPDFを受け付け、`GET /api/uploads` と `GET /api/uploads/{upload_id}` で履歴を確認する。

`DELETE /api/uploads/{upload_id}` はアップロード履歴を論理削除し、保存済みPDF原本をストレージから即削除する。

解析失敗時はアップロード履歴を失敗状態に更新し、監査ログへ `upload.failed` を記録する。

Backend Integration Test では、成功時の明細作成・重複排除・原本削除に加えて、解析失敗時に失敗履歴だけが残り明細を作らないことも確認する。

## 拡張方針

- PDFフォーマットごとに抽出ルールを分ける。
- 解析結果はドメイン明細へ直接保存せず、アプリケーション層で検証してから登録する。
- 抽出できなかった行は失敗理由とともに確認できるようにする。
- 重複明細の取込を防ぐ。
