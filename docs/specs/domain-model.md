# ドメインモデル定義

## 方針

本プロジェクトはドメイン駆動設計（DDD）を採用する。

家計簿アプリの中心はUIやデータベースではなく、家計管理に関する業務ルールである。支出、収入、明細、カテゴリ、アップロード、集計、レポートなどの概念をドメインモデルとして表現し、アプリケーションの振る舞いはドメインを中心に組み立てる。

## 集約

### 明細

ユーザーの支出または収入の1取引を表す。

- 識別子: 明細ID
- 所有者: ユーザーID
- 値: 日付、店名、カテゴリ、金額、支払い方法、利用者、メモ、種別
- 不変条件:
  - 日付は必須
  - 店名は必須
  - カテゴリは必須
  - 金額は整数JPY
  - 金額がマイナスの場合は取消明細を表す
  - 0円明細を許可する
  - 種別は支出または収入

### カテゴリ

明細を分類するための項目を表す。

- 識別子: カテゴリID
- 所有者: ユーザーID
- 値: カテゴリ名、色、説明、有効状態
- 不変条件:
  - カテゴリ名は必須
  - 同一ユーザー内でカテゴリ名は重複しない
  - 使用中カテゴリは物理削除しない

### アップロード履歴

PDF明細のアップロードと解析結果を表す。

- 識別子: アップロードID
- 所有者: ユーザーID
- 値: ファイル名、アップロード日時、取込件数、ステータス、エラー理由
- 不変条件:
  - ファイル名は必須
  - ステータスは処理中、完了、失敗のいずれか
  - 取込件数は0以上

### 監査ログ

明細編集、明細削除、アップロード失敗などの重要な操作や失敗を記録する。

- 識別子: 監査ログID
- 所有者: ユーザーID
- 値: 操作種別、対象リソース種別、対象リソースID、詳細、作成日時

## 値オブジェクト

- 金額: 整数JPY。マイナスの場合は取消明細を表し、0円も許可する
- 日付: `Asia/Tokyo` を基準にした日付
- 日時: DBに保存する日時
- 店名: 明細の取引先名
- カテゴリ名: ユーザーが識別できる分類名
- 支払い方法: クレジットカード、Suica、現金など
- 利用者: カード明細上の利用者
- ファイル名: アップロードされたPDFファイル名

## ドメインサービス候補

- PDF明細の重複判定
- 店名やメモに基づくカテゴリ推定
- 楽天カード明細PDFのルールベース抽出
- 過去分類に基づくカテゴリ自動分類
- 月次集計
- カテゴリ別集計
