# ドメインモデル定義

## 方針

本プロジェクトはドメイン駆動設計（DDD）を採用する。

家計簿アプリの中心はUIやデータベースではなく、家計管理に関する業務ルールである。支出、収入、明細、カテゴリ、アップロード、集計、レポートなどの概念をドメインモデルとして表現し、アプリケーションの振る舞いはドメインを中心に組み立てる。

## 集約

### 明細

ユーザーの支出または収入の1取引を表す。

- 識別子: 明細ID
- 所有者: ユーザーID
- 値: 日付、店名、カテゴリ、金額、支払い方法、メモ、種別
- 不変条件:
  - 日付は必須
  - 店名は必須
  - カテゴリは必須
  - 金額は0以上
  - 種別は支出または収入

### カテゴリ

明細を分類するための項目を表す。

- 識別子: カテゴリID
- 所有者: ユーザーID
- 値: カテゴリ名、色、説明、有効状態
- 不変条件:
  - カテゴリ名は必須
  - 同一ユーザー内でカテゴリ名は重複しない
  - 使用中カテゴリは物理削除しない

### アップロード履歴

PDF明細のアップロードと解析結果を表す。

- 識別子: アップロードID
- 所有者: ユーザーID
- 値: ファイル名、アップロード日時、取込件数、ステータス、エラー理由
- 不変条件:
  - ファイル名は必須
  - ステータスは処理中、完了、失敗のいずれか
  - 取込件数は0以上

## 値オブジェクト

- 金額: 0以上の日本円金額
- 日付: `Asia/Tokyo` を基準にした日付
- 日時: DBに保存する日時
- 店名: 明細の取引先名
- カテゴリ名: ユーザーが識別できる分類名
- 支払い方法: クレジットカード、Suica、現金など
- ファイル名: アップロードされたPDFファイル名

## ドメインサービス候補

- PDF明細の重複判定
- 店名やメモに基づくカテゴリ推定
- 楽天カード明細PDFのルールベース抽出
- 月次集計
- カテゴリ別集計

## 確認事項

- 収入明細を支出明細と同一集約で扱うか。
- 使用中カテゴリ削除時は無効化と未分類付け替えのどちらにするか。
- 重複明細の判定キーをどうするか。
